# Performance + Light-Mode + Cleanup pass — 2026-06-30

A self-contained audit/cleanup pass. Everything below is committed; `tsc --noEmit` is clean
and the pure-logic comps test suite passes. The `tests/` suite (wallet/rarity) runs via `tsx`,
which couldn't execute in the build sandbox — worth a local `npm test` run to confirm green.

## Shipped

### Data / performance (pull less, run smoother)
- **In-flight request coalescing** in both upstream caches (`mintgarden/client.ts`, `market/dexie.ts`).
  When many callers miss the same cache key at once (e.g. a wall of cards needing the same
  collection detail, or two visitors opening the same NFT), they now share ONE network call
  instead of each firing their own. Biggest "pull less data" win under load — kills thundering-herd
  traffic on cache gaps. Failures clear the in-flight entry so retries still work.
- **HTTP cache headers** on the two user-agnostic GET routes:
  - `/api/collection/[id]/all` -> `public, max-age=120, stale-while-revalidate=600`
  - `/api/collections/search` -> `public, max-age=120, stale-while-revalidate=300`
  Repeat views can serve from browser/CDN cache and skip the handler. Both already have
  server-side caches, so data stays fresh.

### Light mode (washed-out fixes)
- `ConfidenceChip` and `CollectionCard` were near-invisible on the white light theme
  (Tailwind `/15` tints + `*-400` text / `white/[0.03]` glass). Both now theme-aware
  (`useThemeMode`) with deepened colors on light, original look preserved on dark.
- Deepened faint light-mode borders: binder collection picker rows, NFT detail modal divider.

## Deferred (valuable but need runtime testing first)
From the data audit. Left alone because they change behavior/accuracy and I can't load-test from here.
1. **Floor resolved twice** on cold cache: `resolveCollectionFloorXch()` (floor + sale-floor) AND
   `fetchCollectionActiveOffers()` both derive the floor. Could have the offers scan return the
   floor as metadata and skip the redundant calls. MEDIUM win; floor accuracy is load-bearing.
2. **`MAX_NFTS = 300`** detail fetches per comps build (compsService.ts). Likely 100-150 suffices
   and would speed cold builds ~2-3x. Needs an accuracy A/B before changing.
3. **Sequential full-collection pagination** (liveCollection.ts): big collections page one-at-a-time
   on first load. Batched parallel paging cuts cold wall-clock ~5x but risks MintGarden rate limits.
4. **Shared cache (Redis/Upstash)** when multi-instance — today each process has its own in-memory
   cache, so coalescing/TTL wins are per-instance.
